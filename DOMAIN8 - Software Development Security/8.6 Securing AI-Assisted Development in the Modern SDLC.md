## 8.6.1 The evolution of coding ##

There was a time when writing code meant understanding every line you produced. Today, that certainty is fading. Developers increasingly rely on AI-assisted coding tools that can generate entire functions, suggest optimizations, and even design architecture patterns. Productivity has skyrocketed—but so has ambiguity.

In this new reality, developers are no longer the sole authors of their code. They are collaborators with an invisible partner: the AI. And like any partner, that AI can make mistakes—subtle, convincing, and sometimes dangerous.

> ℹ The challenge is no longer just securing code written by humans, but also validating code generated by machines. This introduces a new category of risk: not traditional bugs, but “hallucinated” vulnerabilities.

**AI models do not truly understand code; they predict it**. They generate outputs based on patterns learned from vast datasets. Most of the time, this works remarkably well. But occasionally, the AI produces code that looks correct, compiles successfully, and even passes basic tests—yet contains hidden security flaws. These flaws are particularly dangerous because they carry an illusion of authority. Developers may trust the output simply because it was generated quickly and appears sophisticated.

This is where the concept of “insecure by suggestion” emerges. An AI tool might recommend outdated cryptographic functions, improper input validation, or unsafe dependencies. In a fast-paced development environment, these suggestions can be accepted without sufficient scrutiny, embedding vulnerabilities directly into the application from the very beginning.

To counter this, security must shift left—further than ever before. It is no longer enough to test code at the end of the development cycle. AI-generated code must be validated continuously, as it is created. This is achieved by integrating automated security testing directly into the CI/CD pipeline.

Modern pipelines must include tools capable of analyzing both traditional code and AI-generated components. Static Application Security Testing (SAST), Dynamic Application Security Testing (DAST), and Software Composition Analysis (SCA) must be enhanced with AI-aware capabilities. These systems should not only detect known vulnerabilities but also flag suspicious patterns that may originate from machine-generated code.

The goal is simple: catch flaws before they reach production. But the execution requires discipline. Every commit, every merge, every deployment must pass through automated security gates. AI accelerates development, but without automated controls, it also accelerates risk.

Beyond the code itself lies another critical concern: the software supply chain. Modern applications are built on layers of dependencies—libraries, frameworks, and APIs. In the context of AI, this includes machine learning libraries, pre-trained models, and third-party datasets. Each of these components introduces potential vulnerabilities.

The risk here is not theoretical. If a malicious or compromised library is introduced into the development environment, it can affect every application that depends on it. This is particularly dangerous in AI systems, where models and frameworks are often treated as trusted building blocks.

Two emerging threats highlight the importance of securing this layer: model hijacking and inference attacks.

Model hijacking occurs when an attacker gains control over a model or manipulates its behavior. This can happen through compromised dependencies, unauthorized modifications, or weaknesses in deployment pipelines. The result is an AI system that behaves differently than intended—potentially exposing data, making incorrect decisions, or serving malicious outputs.

Inference attacks, on the other hand, target the interaction with the model. By carefully analyzing outputs, an attacker can infer sensitive information about the training data or the model itself. This is not an attack on the infrastructure, but on the logic and behavior of the system.

These threats reinforce a key principle: security must extend beyond the application layer into the entire ecosystem that supports it. Developers must understand not only the code they write, but also the components they integrate.

Embedding AI considerations into the Software Development Life Cycle is therefore essential. Security is no longer a phase; it is a continuous process that spans design, development, testing, deployment, and maintenance. AI tools must be evaluated, monitored, and governed just like any other critical component.

At the same time, organizations must foster a culture of skepticism. Developers should not blindly trust AI-generated outputs. Instead, they should treat them as suggestions—useful, but requiring validation. Secure coding practices remain fundamental, regardless of how the code is produced.

Ultimately, the goal is not to resist AI, but to harness it responsibly. AI-assisted development offers undeniable advantages in speed and efficiency. It allows teams to build more, faster, and with fewer resources. But without proper controls, it also introduces new and complex risks.
